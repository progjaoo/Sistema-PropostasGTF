# PRD — Sistema de Gerenciamento e Criação de Propostas Comerciais
**Versão:** 1.2  
**Data:** Junho 2026  
**Baseado em:** Análise funcional do `radio-proposta_v2.html`  
**Changelog v1.2:** Adição do design system Genesis (seção 18) + Sistema de Templates de Proposta por Categoria (seção 19 + atualizações em 3, 5, 6, 7, 9)

---

## 1. Visão Geral

### 1.1 Contexto

O HTML analisado é um gerador de propostas comerciais para rádio (Rádio 88 FM), com painel lateral de edição e preview A4 paginado em tempo real. O usuário preenche campos como nome da rádio, slogan, tipo de proposta, dados do cliente, período da campanha, 4 estatísticas de audiência, produtos/planos de ação (com cor, quantidade, título, descrição, detalhe, programa e tags), investimento total e dados de contato. A proposta é exportada via `window.print()`. Imagens de logo e banner são carregadas como base64 client-side.

### 1.2 Problema

O HTML standalone possui limitações críticas:

- **Sem persistência:** cada proposta é descartada ao fechar o navegador.
- **Sem controle de acesso:** qualquer pessoa com o arquivo pode gerar propostas da empresa.
- **Sem histórico:** impossível auditar, reabrir ou editar uma proposta enviada.
- **Sem gestão de clientes/empresas:** dados de cliente são digitados do zero a cada vez.
- **Sem multi-usuário:** um único arquivo para toda a equipe comercial.
- **Sem templates reutilizáveis:** configurações de produtos precisam ser refeitas manualmente.
- **Sem templates de proposta por categoria:** não há modelos prontos por setor (Veicular, Varejo, etc.).

### 1.3 Solução

Sistema web full-stack com autenticação JWT, gestão de usuários por um Admin Master, cadastro de empresas anunciantes, templates de produtos, **templates completos de proposta organizados por categoria de negócio**, criação/edição/histórico de propostas e exportação PDF. O layout visual da proposta é fiel ao HTML original. A interface do sistema segue o **design system Genesis** (indigo + tipografia editorial).

## 2. Objetivos do Produto

| # | Objetivo | Métrica de Sucesso |
|---|----------|--------------------|
| 1 | Centralizar criação e histórico de propostas | 100% das propostas salvas no banco |
| 2 | Controlar quem pode criar propostas | 0 propostas geradas por usuários não autorizados |
| 3 | Agilizar o processo comercial | Redução de ≥50% no tempo de montagem de uma proposta |
| 4 | Reutilizar configurações de produtos | Criação de proposta a partir de template em < 2 min |
| 5 | Manter identidade visual da rádio | Preview A4 idêntico ao HTML original |
| 6 | Acelerar propostas por setor | Proposta completa a partir de template de categoria em < 1 min |

## 3. Personas

### 3.1 Admin Master
- Único usuário com acesso total ao sistema.
- Gerencia usuários (criação, ativação/desativação, reset de senha).
- Gerencia o cadastro da(s) rádio(s)/empresa(s) emissora(s) e Empresas Anunciantes.
- Acessa todas as propostas de todos os usuários.
- Define templates globais de produtos.
- **Cria e mantém os Templates de Proposta por Categoria** (Veicular, Varejo, Saúde, etc.).

### 3.2 Usuário Comercial (criado pelo Admin)
- Cria, edita e visualiza suas próprias propostas.
- Seleciona empresas anunciantes já cadastradas.
- **Cria propostas a partir de templates de categoria com um clique.**
- Exporta PDF das propostas.
- Não acessa configurações de sistema nem outros usuários.

## 4. Stack Técnica

| Camada | Tecnologia |
|--------|-----------|
| Backend | Node.js + Express (ou Fastify) |
| ORM | Prisma ORM |
| Banco | PostgreSQL |
| Autenticação | JWT (access token 15min + refresh token 7d, httpOnly cookie) |
| Frontend | React 18+ + Vite |
| UI | Tailwind CSS + shadcn/ui + **Genesis Design System** |
| PDF Export | `react-to-print` ou `html2pdf.js` client-side |
| Upload imagem | Base64 (armazenado no banco como `TEXT`) |
| Containerização | Docker Compose (Postgres + API + Frontend) |
| Alertas/Feedback | react-toastify — Modais em cadastros sensíveis (deletar, atualizar) |
| Máscaras/Validação | Máscaras em valores monetários, CNPJ, CPF, telefone; validação com Zod |

generator client {
  provider = "prisma-client-js"

  provider = "postgresql"

  id        String   @id @default(cuid())

  userId    String

  expiresAt DateTime
  createdAt DateTime @default(now())
}

// ─────────────────────────────────────────────
// EMISSORAS (entidade dona do sistema)
// ─────────────────────────────────────────────

model Station {
  id        String   @id @default(cuid())

  slogan    String?                     // "A sua rádio"

  createdAt DateTime @default(now())

  proposals        Proposal[]
  productTemplates ProductTemplate[]
  proposalTemplates ProposalTemplate[]  // ← NOVO
}

// EMPRESAS ANUNCIANTES (clientes da rádio)
// ─────────────────────────────────────────────

model Advertiser {
  id           String   @id @default(cuid())

  logoBase64   String?  @db.Text

  contactPhone String?

  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  proposals Proposal[]
}

// ─────────────────────────────────────────────
// TEMPLATES DE PRODUTO (reutilizáveis)
// ─────────────────────────────────────────────

model ProductTemplate {
  id        String   @id @default(cuid())
  stationId String
  station   Station  @relation(fields: [stationId], references: [id])
  name      String                   // "Testemunhal Fato Popular"
  qty       String   @default("01")
  title     String
  description String? @db.Text
  detail    String?
  program   String?
  tags      String[] @default([])
  color     ProductColor @default(BLUE)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

  BLUE
  YELLOW
  RED
  GREEN
  DARK
}

// ─────────────────────────────────────────────
// CATEGORIAS DE TEMPLATE DE PROPOSTA ← NOVO
// ─────────────────────────────────────────────

model ProposalCategory {
  id          String   @id @default(cuid())
  name        String                       // "Empresa Veicular", "Varejo", "Saúde"
  slug        String   @unique             // "veicular", "varejo", "saude"
  description String?
  icon        String?                      // nome do ícone (lucide) ou emoji
  active      Boolean  @default(true)
  order       Int      @default(0)         // ordem de exibição
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  templates   ProposalTemplate[]
}

// ─────────────────────────────────────────────
// TEMPLATES DE PROPOSTA POR CATEGORIA ← NOVO
// ─────────────────────────────────────────────

model ProposalTemplate {
  id          String           @id @default(cuid())
  stationId   String
  station     Station          @relation(fields: [stationId], references: [id])
  categoryId  String
  category    ProposalCategory @relation(fields: [categoryId], references: [id])
  name        String           // "Pacote Lançamento Veicular", "Promoção Sazonal Varejo"
  description String?
  active      Boolean          @default(true)
  createdAt   DateTime         @default(now())
  updatedAt   DateTime         @updatedAt

  // Snapshot dos campos da proposta pré-preenchidos
  propType    String           @default("Proposta Comercial")
  campTag     String?
  periodDesc  String?
  investDesc  String?
  stats       Json             // [{num, suf, desc}] — valores de exemplo da categoria
  overlayOpacity Int           @default(70)

  products    ProposalTemplateProduct[]
  proposals   Proposal[]               // propostas geradas a partir deste template
}

model ProposalTemplateProduct {
  id          String           @id @default(cuid())
  templateId  String
  template    ProposalTemplate @relation(fields: [templateId], references: [id], onDelete: Cascade)
  order       Int              @default(0)
  qty         String           @default("01")
  title       String
  description String?          @db.Text
  detail      String?
  program     String?
  tags        String[]         @default([])
  color       ProductColor     @default(BLUE)
}

// ─────────────────────────────────────────────
// PROPOSTAS
// ─────────────────────────────────────────────

enum ProposalStatus {
  DRAFT       // Rascunho
  SENT        // Enviada ao cliente
  APPROVED    // Aprovada
  REJECTED    // Recusada
  ARCHIVED    // Arquivada

model Proposal {
  id           String         @id @default(cuid())
  stationId    String
  station      Station        @relation(fields: [stationId], references: [id])
  advertiserId String?

  createdById  String
  createdBy    User           @relation(fields: [createdById], references: [id])
  status       ProposalStatus @default(DRAFT)

  // Origem: gerada a partir de um template de categoria ← NOVO
  fromTemplateId String?
  fromTemplate   ProposalTemplate? @relation(fields: [fromTemplateId], references: [id])

  // Dados da proposta (espelho do HTML)
  propType     String         @default("Proposta Comercial")
  propMonth    String
  propYear     String
  campTag      String?
  clientLine1  String?
  clientLine2  String?
  dateStart    String?
  dateEnd      String?
  periodDesc   String?
  bannerBase64 String?  @db.Text
  overlayOpacity Int   @default(70)

  stats        Json            // [{num, suf, desc}]

  investDesc   String?
  investValue  String?

  contactName  String?
  contactRole  String?
  contactPhone String?

  products     ProposalProduct[]
  versions     ProposalVersion[]
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}

model ProposalProduct {
  id          String       @id @default(cuid())
  proposalId  String
  proposal    Proposal     @relation(fields: [proposalId], references: [id], onDelete: Cascade)
  order       Int          @default(0)
  qty         String       @default("01")
  title       String
  description String?      @db.Text
  detail      String?
  program     String?
  tags        String[]     @default([])

  id         String   @id @default(cuid())

  proposal   Proposal @relation(fields: [proposalId], references: [id], onDelete: Cascade)

  createdById String?

## 6. API REST — Endpoints

### Autenticação

| Método | Rota | Descrição |
|--------|------|-----------|
| POST | `/auth/login` | Login com email + senha → access token + refresh token (cookie) |
| POST | `/auth/refresh` | Renova access token via refresh token |
| POST | `/auth/logout` | Invalida refresh token |
| GET  | `/auth/me` | Dados do usuário autenticado |

### Usuários (Admin only)

| Método | Rota | Descrição |
|--------|------|-----------|
| GET | `/users` | Listar usuários |
| POST | `/users` | Criar usuário |
| PATCH | `/users/:id` | Editar nome, email, role, status |
| POST | `/users/:id/reset-password` | Admin reseta senha de um usuário |
| DELETE | `/users/:id` | Desativar usuário (soft delete) |

### Emissora / Station (Admin only)

| Método | Rota | Descrição |
|--------|------|-----------|
| GET | `/stations` | Listar emissoras |
| POST | `/stations` | Criar emissora |
| PATCH | `/stations/:id` | Editar (nome, slogan, logo base64) |

### Empresas Anunciantes

| Método | Rota | Descrição |
|--------|------|-----------|
| GET | `/advertisers` | Listar anunciantes (com busca/filtro) |
| POST | `/advertisers` | Criar anunciante |
| GET | `/advertisers/:id` | Detalhe de um anunciante |
| PATCH | `/advertisers/:id` | Editar anunciante |
| DELETE | `/advertisers/:id` | Desativar (soft delete) |

### Templates de Produto (Admin)

| Método | Rota | Descrição |
|--------|------|-----------|
| GET | `/product-templates` | Listar templates |
| POST | `/product-templates` | Criar template |
| PATCH | `/product-templates/:id` | Editar template |
| DELETE | `/product-templates/:id` | Remover template |

### Categorias de Template ← NOVO

| Método | Rota | Descrição |
|--------|------|-----------|
| GET | `/proposal-categories` | Listar categorias (público p/ usuários autenticados) |
| POST | `/proposal-categories` | Criar categoria (Admin) |
| PATCH | `/proposal-categories/:id` | Editar categoria (Admin) |
| DELETE | `/proposal-categories/:id` | Remover categoria (Admin) |

### Templates de Proposta por Categoria ← NOVO

| Método | Rota | Descrição |
|--------|------|-----------|
| GET | `/proposal-templates` | Listar todos os templates (com filtro por `categoryId`) |
| GET | `/proposal-templates/:id` | Detalhe de um template (com produtos) |
| POST | `/proposal-templates` | Criar template (Admin) |
| PATCH | `/proposal-templates/:id` | Editar template (Admin) |
| DELETE | `/proposal-templates/:id` | Remover template (Admin) |
| POST | `/proposal-templates/:id/use` | **Criar uma proposta a partir do template** — retorna nova Proposal |

### Propostas

| Método | Rota | Descrição |
|--------|------|-----------|
| GET | `/proposals` | Listar propostas do usuário (Admin vê todas) — com paginação e filtros |
| POST | `/proposals` | Criar proposta (cria versão inicial automaticamente) |
| GET | `/proposals/:id` | Detalhe completo de uma proposta |
| PATCH | `/proposals/:id` | Salvar alterações (salva nova versão automaticamente) |
| PATCH | `/proposals/:id/status` | Alterar status (DRAFT → SENT → APPROVED/REJECTED) |
| GET | `/proposals/:id/versions` | Listar versões (histórico) |
| GET | `/proposals/:id/versions/:versionId` | Restaurar snapshot de versão |
| DELETE | `/proposals/:id` | Arquivar/excluir proposta |

## 8. Fluxos Funcionais Principais

### 8.1 Autenticação

```
[Login Page]
  → POST /auth/login (email + senha)
  ← access_token (bearer) + refresh_token (httpOnly cookie)
  → Armazena access_token em memória (Zustand)
  → Redireciona para /dashboard

  [Token expirado]
  → Interceptor Axios detecta 401
  → POST /auth/refresh automático
  → Retry da requisição original
```

### 8.2 Criar Proposta (fluxo atualizado)

```
[Dashboard] → "Nova Proposta"
  → Exibe CategoryGrid (grade de categorias: Veicular, Varejo, Saúde, etc.)
  → Usuário clica em uma categoria → abre TemplatePicker com cards dos templates
  → Usuário escolhe um template (ou "Em branco")
  → POST /proposal-templates/:id/use → retorna Proposal rascunho com campos pré-preenchidos
  → Abre ProposalEditor + ProposalPreview com o conteúdo do template já carregado
  → Usuário ajusta campos específicos do cliente
  → "Salvar Rascunho" → PATCH /proposals/:id
  → "Exportar PDF" → ProposalPrintView → window.print()
  → "Marcar como Enviada" → PATCH /proposals/:id/status
```

### 8.3 Gerenciar Usuários (Admin)

```
[Admin > Usuários] → "Novo Usuário"
  → Preenche: Nome, Email, Senha temporária, Role
  → POST /users
  → Usuário recebe acesso com a senha definida pelo Admin
  → Admin pode: Editar, Ativar/Desativar, Resetar Senha
```

### 8.4 Cadastro de Anunciante

```
[Anunciantes] → "Novo Anunciante"
  → Preenche: Nome fantasia, Razão social, CNPJ, Logo (upload → base64),
              Contato (nome, telefone, email), Observações
  → POST /advertisers
  → Disponível para seleção em qualquer proposta
```

### 8.5 Criar Template de Proposta por Categoria (Admin) ← NOVO

```
[Admin > Templates de Proposta] → "Novo Template"
  → Seleciona Categoria (ex: "Empresa Veicular")
  → Preenche: Nome do template, Descrição
  → Preenche campos pré-definidos: tipo de proposta, tag da campanha, período desc,
    estatísticas de exemplo (placeholder para o setor), investimento desc
  → Adiciona produtos (igual ao editor de proposta: cor, qtd, título, etc.)
    pode importar de ProductTemplates existentes
  → Salva → template disponível para todos os usuários comerciais
```

## 9. Telas do Sistema

### 9.1 Login
- Campos: Email + Senha
- Botão "Entrar"
- Feedback de erro inline (credenciais inválidas, conta inativa)
- Sem cadastro público — apenas Admin cria contas

### 9.2 Dashboard
- Cards de resumo: Total de propostas, Enviadas, Aprovadas, Recusadas
- Lista das últimas 5 propostas do usuário (com status badge colorido)
- Botão "Nova Proposta" em destaque → inicia pelo CategoryGrid
- Admin vê resumo global + acesso rápido a gerenciamento

### 9.3 Lista de Propostas
- Tabela com: Nº/ID, Anunciante, Tipo, Mês/Ano, Status, Template de Origem, Criado por, Data
- Filtros: Status, Período, Anunciante, Categoria, Usuário (Admin)
- Busca por texto livre
- Paginação server-side
- Ações por linha: Editar, Duplicar, Exportar PDF, Arquivar

### 9.4 Nova Proposta — Seleção de Categoria e Template ← NOVO

**Tela intermediária antes de abrir o editor:**

- **Grade de Categorias** (CategoryGrid): cards visuais com ícone e nome da categoria
  - Ex: 🚗 Empresa Veicular · 🛒 Varejo · 🏥 Saúde · 🏗️ Construção · 📋 Em branco
- Clique em uma categoria → **TemplatePicker** (modal ou painel lateral) com os templates disponíveis
- Cada TemplateCard exibe: nome do template, descrição resumida, lista dos produtos inclusos (chips/tags), badge da categoria
- Botão "Usar este template" → cria proposta rascunho com campos pré-preenchidos e redireciona ao editor
- Botão "Em branco" sempre disponível → abre o editor sem pré-preenchimento

### 9.5 Editor de Proposta (tela principal)
Layout split 380px + resto (fiel ao HTML original):

**Painel Esquerdo (Editor):**
- Seção: Emissora (nome, slogan, logo upload)
- Seção: Proposta (tipo, mês/ano)
- Seção: Campanha (tag, cliente linha 1 e 2)
- Seção: Período (data início, data fim, descrição)
- Seção: Banner (upload imagem, opacidade overlay)
- Seção: Estatísticas (4 blocos: número, sufixo, descrição)
- Seção: Plano de Ações (lista de produtos)
  - Campos por produto: quantidade, título, descrição, detalhe, programa, tags, cor
  - Botão "Adicionar Produto" / "Importar do Template" / "Importar do Template de Categoria"
- Seção: Investimento (descrição, valor)
- Seção: Contato (nome, cargo, telefone)
- **Badge informativo quando a proposta foi gerada de um template** (ex: "Baseada em: Pacote Lançamento Veicular")

**Painel Direito (Preview A4):**
- Preview em tempo real, com paginação automática
- Fiel ao layout visual do HTML (cores, tipografia, grid de produtos)
- Label "PREVIEW" sobre o documento

**Footer do Editor:**
- Botão "Salvar Rascunho" (auto-save a cada 30s também)
- Botão "Exportar PDF"
- Dropdown de status

### 9.6 Anunciantes
- Lista com logo, nome, CNPJ, contato, status
- Formulário de criação/edição com upload de logo (base64)
- Histórico de propostas do anunciante (read-only)

### 9.7 Admin — Usuários

## 10. Regras de Negócio

| Regra | Descrição |
|-------|-----------|
| RN-01 | Apenas Admin cria, edita e desativa usuários |
| RN-02 | Usuário COMERCIAL só visualiza e edita suas próprias propostas |
| RN-03 | Admin visualiza e pode editar qualquer proposta |
| RN-04 | Proposta com status APPROVED ou REJECTED não pode ser editada sem reverter para DRAFT (apenas Admin) |
| RN-05 | A cada PATCH em uma proposta, uma nova ProposalVersion é criada automaticamente |
| RN-06 | Máximo de 50 versões por proposta (purge automático das mais antigas) |
| RN-07 | Logo e banner são armazenados como base64 (TEXT) no banco — sem storage externo |
| RN-08 | Limite recomendado para imagens: 2MB antes do encode (≈ 2.7MB em base64) |
| RN-09 | CNPJ do anunciante é único no sistema |
| RN-10 | Usuário inativo não consegue fazer login |
| RN-11 | Refresh token é invalidado no logout e substituído a cada renovação (rotation) |
| RN-12 | Senha mínima: 8 caracteres |
| RN-13 | Apenas Admin cria, edita e desativa Categorias e Templates de Proposta ← NOVO |
| RN-14 | A criação de proposta via template (`POST /proposal-templates/:id/use`) registra `fromTemplateId` na proposta gerada ← NOVO |
| RN-15 | Ao usar um template, os campos específicos do cliente (clientLine1, clientLine2, dateStart, dateEnd, investValue, bannerBase64) são deixados em branco para preenchimento manual — apenas campos estruturais são copiados ← NOVO |
| RN-16 | Editar ou excluir um ProposalTemplate não afeta propostas já geradas a partir dele ← NOVO |

## 11. Segurança

- Senhas com `bcrypt` (salt rounds ≥ 12)
- JWT access token com expiração curta (15 min)
- Refresh token em httpOnly + Secure cookie
- Refresh token rotation
- Rate limiting em `/auth/login` (5 tentativas / 15 min por IP)
- Todos os endpoints autenticados exigem `Authorization: Bearer <token>`
- Middleware de role: ADMIN_ONLY para rotas de gestão e de templates de categoria
- Validação de schema no backend com Zod
- CORS configurado para domínio do frontend apenas

- Fonte Barlow Condensed / Barlow via Google Fonts (ou embed local)
- Paginação automática replicada via `utils/pagination.ts`
- `@media print` CSS para remover o editor e exibir apenas as páginas A4
- Opção preferencial: `react-to-print`
- Fallback: `html2pdf.js` com `scale: 2`

## 18. Design System — Genesis ← NOVO

O frontend do sistema adota o **Genesis Design System** (`https://designmd.ai/chef/genesis`) como base visual, garantindo uma interface editorial, limpa e confiante alinhada com o posicionamento comercial da rádio.

### 18.1 Identidade Visual do Sistema (interface, não da proposta)

> A proposta gerada mantém sua própria identidade visual (Barlow Condensed, cores da rádio). O Genesis se aplica exclusivamente à interface do sistema (sidebar, formulários, dashboard, listas, modais).

### 18.2 Paleta de Cores

| Token | Valor | Uso |
|-------|-------|-----|
| `--color-primary` | `#6366F1` | CTAs, estados ativos, links, focus rings, destaques interativos (indigo) |
| `--color-primary-hover` | `#4F46E5` | Hover em elementos primários |
| `--color-neutral` | `#9C9C9C` | Textos secundários, placeholders, timestamps, estados desabilitados |
| `--color-background` | `#FAFAFA` | Fundo das páginas (cinza claro quente) |
| `--color-surface` | `#FFFFFF` | Cards, painéis, modais, nav |
| `--color-text-primary` | `#0A0A0A` | Headings, body, labels principais (quase-preto) |
| `--color-text-secondary` | `#6B6B6B` | Descrições, metadados, labels secundários |
| `--color-border` | `#E8E8EC` | Bordas de cards, divisores, inputs (sutil e recessivo) |
| `--color-success` | `#10B981` | Status Aprovada, confirmações |
| `--color-warning` | `#F59E0B` | Status Pendente, alertas |
| `--color-error` | `#EF4444` | Ações destrutivas, erros de validação, status Recusada |

### 18.3 Tipografia

| Escala | Fonte | Tamanho | Peso | Uso |
|--------|-------|---------|------|-----|
| Display | General Sans | 72px | Bold | Títulos de marketing (apenas landing/onboarding) |
| Headline | General Sans | 60px | Bold | Títulos de página grandes |
| Section Heading | General Sans | 32px | Bold | Títulos de seção no dashboard |
| Subhead | General Sans | 24px | Bold | Subtítulos, nomes de blocos |
| Body | DM Sans | 15px | Regular | Texto corrido, descrições |
| Small | DM Sans | 13px | Regular | Metadados, labels de formulário |
| Caption | DM Sans | 12px | Regular | Tooltips, notas de rodapé |
| Overline | DM Sans | 11px | Regular | Labels superiores, categorias |

**Regra:** General Sans para headings, DM Sans para body — nunca trocar entre si.

### 18.4 Espaçamento

Base: **4px**. Escala: 4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96px. Todas as margens, paddings e gaps devem respeitar múltiplos de 4px.

### 18.5 Componentes Genesis

**Botões:**
- Primário (filled indigo `#6366F1`): apenas 1 por seção de view — ações principais como "Salvar", "Nova Proposta", "Criar Template"
- Secundário (outlined): ações de suporte como "Cancelar", "Voltar"
- Radius: **6px** para botões e inputs

**Cards:**
- Radius: **12px**
- Superfície: `#FFFFFF` com borda `#E8E8EC`
- Sombra: apenas em hover e focus states (não em elementos estáticos)

**Inputs:**
- Radius: **6px**
- Borda: `#E8E8EC` em repouso, `#6366F1` em focus
- Mensagem de erro: `#EF4444`

**Status Badges (propostas):**
- DRAFT → neutro (`#9C9C9C`)
- SENT → warning (`#F59E0B`)
- APPROVED → success (`#10B981`)
- REJECTED → error (`#EF4444`)
- ARCHIVED → neutro escuro

### 18.6 Regras Do's & Don'ts

**Fazer:**
- Usar indigo (`#6366F1`) exclusivamente para elementos interativos
- Manter grid de espaçamento de 4px em todos os componentes
- General Sans em headings, DM Sans em body — sem exceções
- Radius 12px em cards, 6px em botões/inputs — não misturar
- Garantir contraste adequado em modo claro

**Não fazer:**
- Usar preto puro (`#000000`) ou branco puro (`#FFFFFF`) para textos — usar os tokens definidos
- Adicionar gradientes decorativos ou ilustrações desnecessárias
- Aplicar sombras em elementos estáticos — reservar para hover e focus
- Usar mais de dois pesos de fonte numa mesma tela
- Colocar mais de um botão primário (filled indigo) na mesma seção de view

## 19. Sistema de Templates de Proposta por Categoria ← NOVO

### 19.1 Visão Geral

Templates de proposta são modelos completos e reutilizáveis, organizados por categoria de negócio do anunciante. Eles eliminam o trabalho repetitivo de configurar produtos e campos estruturais a cada nova proposta para um segmento conhecido.

**Analogia:** assim como um escritório de advocacia tem minutas por tipo de contrato, o time comercial da rádio terá modelos prontos por tipo de anunciante.

### 19.2 Estrutura de uma Categoria

Cada categoria representa um segmento de mercado atendido pela rádio:

| Categoria | Slug | Ícone | Exemplos de templates |
|-----------|------|-------|----------------------|
| Empresa Veicular | `veicular` | 🚗 | Pacote Lançamento, Manutenção de Marca, Queima de Estoque |
| Varejo | `varejo` | 🛒 | Promoção Sazonal, Black Friday, Aniversário de Loja |
| Saúde | `saude` | 🏥 | Institucional Clínica, Campanha de Vacinas |
| Alimentação | `alimentacao` | 🍔 | Delivery, Inauguração, Promoção de Cardápio |
| Construção & Imóveis | `construcao` | 🏗️ | Lançamento Imóvel, Construtora Institucional |
| Serviços | `servicos` | 📋 | Educação, Jurídico, Contabilidade |

### 19.3 O que um Template pré-preenche

Quando o usuário escolhe um template, os seguintes campos são copiados para a nova proposta:

| Campo | Copiado? | Observação |
|-------|----------|------------|
| Tipo de proposta | ✅ Sim | Ex: "Proposta Veicular" |
| Tag da campanha | ✅ Sim | Ex: "LANÇAMENTO" |
| Descrição do período | ✅ Sim | Ex: "campanha mensal" |
| Estatísticas (4 blocos) | ✅ Sim | Valores de exemplo do segmento |
| Descrição do investimento | ✅ Sim | Ex: "Investimento mensal" |
| Opacidade do overlay | ✅ Sim | |
| Produtos (lista completa) | ✅ Sim | Copiados como novos ProposalProducts |
| Nome do cliente (linha 1 e 2) | ❌ Não | Preenchimento manual |
| Data início / fim | ❌ Não | Preenchimento manual |
| Valor do investimento | ❌ Não | Preenchimento manual |
| Banner (imagem) | ❌ Não | Upload manual |
| Logo da emissora | ❌ Não | Vem da Station configurada |

### 19.5 Criação de Template pelo Admin

O formulário de criação de template no painel Admin replica o editor de proposta, mas sem os campos específicos do cliente:

1. **Dados gerais:** nome do template, categoria, descrição
2. **Campos estruturais:** tipo de proposta, tag da campanha, período desc, investimento desc, estatísticas de exemplo (4 blocos com valores representativos do segmento)
3. **Produtos:** editor idêntico ao da proposta — adicionar, reordenar, definir cor, qtd, título, descrição, detalhe, programa, tags
4. **Preview A4:** painel de preview ao lado (sem dados de cliente, apenas estrutura)
5. **Salvar como ativo ou rascunho**

### 19.6 Métricas e Análise

O campo `fromTemplateId` na Proposal permite:
- Dashboard Admin: "Templates mais utilizados" (ranking por uso)
- Filtro na lista de propostas: "Ver propostas geradas pelo template X"
- Análise futura: taxa de aprovação por categoria de template
